Machine Learning

1. CatBoost Dosha Prediction and SHAP Explanation

When a user completes the health questionnaire, the Node.js backend spawns a Python child process that loads a pre-trained CatBoost gradient-boosted classifier. The model runs multi-class inference across 25 encoded features, then uses CatBoost's native SHAP (Shapley Additive exPlanations) to compute per-feature contribution scores for explainability.

    START(["Start"]) --> A

    subgraph UserStart ["User"]
        A["Complete all 25 health trait questions via React frontend"]
        B["Submit JSON payload via POST /api/dosha/analyze"]
        A --> B
    end

    B --> C

    subgraph System1 ["System"]
        C["Node.js DoshaAnalyzer receives JSON and spawns Python child process via child_process.execFile"]
        D["Python script loads CatBoostClassifier, LabelEncoders, and metadata from disk using joblib.load"]
        E["Parse input JSON into a pandas DataFrame with 25 feature columns"]
        F["Apply fitted sklearn LabelEncoders to transform each categorical string value into its integer encoding"]
        G{"Any expected features missing from input?"}
        H["Fill missing features with 'Unknown', which encodes to le.classes_ index 0 as fallback"]
        I["Reorder DataFrame columns to match the exact training feature order from model_metadata.json"]
        J["Call model.predict_proba to get softmax probability distribution across 3 classes: Vata, Pitta, Kapha"]
        K["Select argmax of probability array as Primary dosha"]
        LL["Select second-highest via argsort as Secondary dosha"]
        M{"Primary probability >= 60%?"}

        U["Build JSON result: scores, primary, secondary, constitutionType, confidence, xai_insights"]
        V["Print JSON to stdout, Node.js captures and parses it via child process pipe"]

        I --> J --> K --> LL --> M

Key files: catboost_model.py (Python ML pipeline), DoshaAnalyzer.js (Node.js orchestrator)
Model config: CatBoostClassifier with iterations=100, learning_rate=0.1, depth=5, loss_function=MultiClass, random_seed=42.
SHAP: Uses CatBoost's native Shapley value computation (not the separate shap library), which is exact and efficient for tree-based models. Feature contributions are signed values indicating direction and magnitude of influence on the predicted class.
AI Orchestration

2. AI Service Manager - Orchestrator

A Strategy-pattern orchestrator that manages multiple AI provider instances. It performs RAG (Retrieval-Augmented Generation) by injecting Knowledge Base search results into the system prompt, delegates to the active provider, and implements automatic failover with provider metadata tagging on every response.

        C["AIServiceManager.generateResponse receives messages array and doshaProfile"]
        D["Extract last user message, call KnowledgeBase.search with query and topK=3"]
        E{"searchResults.length > 0?"}
        F["Inject RAG context into system message: append formatted herbs, principles, medicines to prompt"]

        K["Call CustomAI.generateResponse with messages and doshaProfile"]
        LL{"Provider returned successfully without throwing?"}
        M["Spread response object with provider name and ISO 8601 timestamp"]
        N["Return enriched response to Express route handler"]
        O{"this.currentProvider !== this.fallbackProvider?"}
        P["Swap this.currentProvider to fallbackProvider and recursively call generateResponse"]
        Q{"Fallback succeeded?"}
        R["Return response with fallbackUsed=true and originalProvider field"]
        S["Restore original provider, throw error to Express error handler"]

Key file: AIServiceManager.js (Strategy pattern orchestrator)
Provider registry: { custom: new CustomAI(), gemini: new GeminiAI() } — instantiated at startup.
RAG injection: Knowledge context is appended to the system role message with the prefix "ADDITIONAL AYURVEDIC KNOWLEDGE (verified from classical texts)" before being passed to the active provider.
Failover: If the primary provider throws, the manager swaps to this.fallbackProvider ('custom') and retries the full pipeline. The CustomAI engine is always available as it requires no external API.
AI Provider

3. Gemini LLM Provider

Interfaces with the Google Generative AI SDK (@google/generative-ai) to call the Gemini 2.0 Flash model. Constructs a multi-turn chat session with a domain-specific system instruction, injects the user's dosha constitution and any RAG context, sanitises conversation history to comply with Gemini's strict user/model alternation requirement, and implements exponential backoff retry for HTTP 429/503 rate limits.

Key file: GeminiAI.js (wraps @google/generative-ai SDK)
Model: gemini-2.0-flash with generationConfig: temperature=0.7, maxOutputTokens=800, topP=0.9, topK=40.
Safety settings: All 4 harm categories set to BLOCK_ONLY_HIGH to allow health/medical discussions.
History sanitisation: Gemini's chat API requires strict user/model alternation — the _sanitiseHistory method merges consecutive same-role messages and strips leading model entries.
Retry: Exponential backoff at 2n+1 seconds (2s → 4s → 8s) for rate-limit errors (HTTP 429, 503, RESOURCE_EXHAUSTED).
AI Provider

4. Rule-Based AI Provider

A deterministic, zero-dependency AI engine that runs entirely on the Node.js server with no external API calls. Uses a keyword-frequency intent classifier to detect 7 topic categories, resolves dosha context from the user profile or message text via string matching, selects from parameterised response templates populated from local data modules (ayurvedic-knowledge.js, food-database.js, research-studies.js), and augments output with a secondary KnowledgeBase.search call for RAG-style enrichment.

Key file: CustomAI.js (390 lines, deterministic engine)
Data modules: ayurvedic-knowledge.js (dosha profiles, yoga practices, mental health mappings), food-database.js (balancing foods), research-studies.js (evidence references).
Intent classification: Keyword-frequency scoring across 7 categories (food, lifestyle, exercise, health, mental, sleep, digestion) — the category with the most keyword hits wins.
Availability: isAvailable() always returns true — this engine is the guaranteed fallback since it has zero external dependencies.
Data Pipeline

5. RAG Knowledge Retrieval

Implements a Retrieval-Augmented Generation pipeline within the AIServiceManager. Before dispatching to any AI provider, the system performs a weighted keyword search across 3 local JSON data stores (704 herbs, 162 principles, 29 medicines), applies dosha-specific relevance boosting, and injects the top-K results into the LLM's system prompt so generated responses are grounded in verified Ayurvedic texts rather than relying solely on the model's parametric knowledge.

Data Layer

6. Knowledge Base Search

A singleton weighted keyword search engine exported as a shared module instance. Implements a TF-IDF-inspired scoring algorithm with tiered weighting (text match, name match, exact match) plus a dosha-specific disambiguation layer that checks each item's pacify field to boost or penalise relevance. All 895 items from 3 JSON stores are loaded lazily on first access and cached in memory for subsequent queries.

Key file: KnowledgeBase.js (105 lines, singleton via module.exports = new KnowledgeBase())
Scoring algorithm: Keyword in text body = +2 pts, keyword in name = +5 pts, exact name match = +10 pts, item pacifies queried dosha = +10 pts, item mentions dosha in text = +3 pts, item unrelated to dosha = −5 pts. Threshold filter removes items scoring ≤ 5.
Output format: formatContext() produces structured strings per type — herbs include Rasa, Virya, Vipaka, Pacify fields; principles include explanation and clinical importance; medicines include botanical name, dosage, and parts used.
