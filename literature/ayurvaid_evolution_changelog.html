AyurVAID Chatbot Evolution

A chronological, stage-by-stage walkthrough detailing the historical development, structural progression, and code transformations of the AyurVAID system.

Chronological Timeline Roadmap

This chronicle walks through the development phases of the AyurVAID AI module, logging the legacy starting states, intermediary trial structures, and the final optimized production-ready implementation.

STAGE 1

Establishing the Architectural Foundation

At its inception, AyurVAID focused on solidifying local, deterministic intelligence and robust Machine Learning structures. This stage built the structural baseline of the Prakriti analysis features and local user frameworks.

Before / Initial State

No health intelligence pathways existed. The application lacked dosha diagnostics, user profile databases, or conversational structures.

After / Completed Infrastructure
  • CatBoost ML Classification: Trained gradient-boosted decision trees on a 5,000-row Prakriti dataset to classify users into Dosha groups.
  • SHAP explainability (xAI): Native SHAP value calculations directly mapped features contributing to Dosha results.
  • Custom Rule Engine (`CustomAI.js`): Built local static keyword intents (digestion, lifestyle, sleep) to guide users offline.
  • Secure Core: Configured JWT validation routes linked to Firebase.
💡 Stage 1 Architectural Purpose

Prioritizing CatBoost and offline rules established high classification performance and reliable safety thresholds before integrating heavy, dynamic natural language elements.

STAGE 2

Multi-Provider AI Expansion & Rasa Integration

To achieve conversational fluidity and contextual NLP interactions, the system was expanded into a multi-provider testbed. This added heavy semantic parsing and external language model layers.

Starting State (Stage 1)

Limited to static keyword heuristics (`CustomAI`) without natural language parsing, entity recognition, or generic LLM intelligence.

Expanded Architecture
  • Rasa NLU Server: Integrated Rasa NLU microservice running character-level n-grams and DIET intent transformers on port :5005.
  • Ollama / LocalAI Integration: Added `LocalAI.js` mapping local LLMs (Mistral/Llama) to support offline conversational needs.
  • HuggingFace Pipelines: Implemented `HuggingFaceAI.js` utilizing DialoGPT configurations.
  • OpenAI Capabilities: Integrated `AIServiceManager.js` to optionally redirect chats to GPT-4.
  • Keyword RAG Pipeline: Built `KnowledgeBase.js` to search 450+ classical herbs, principles, and medicines from static JSON arrays.
💡 Stage 2 Architectural Purpose

This stage targeted dual intent-entity matching. Coupling Rasa (for NLU) with LLMs (for generative text) and offline JSON arrays (for RAG) allowed the system to answer complex Ayurvedic queries, albeit at massive resource costs.

STAGE 3

Migrating to Cloud Gemini LLM & RAG Optimization

Due to high resource saturation and API bottlenecks, the core conversational layer was transitioned to a premium cloud-driven generative API model.

Starting State (Stage 2)

Fragmented LLM layers without native safety settings, rate-limiting handlers, or strict chat memory adaptations, resulting in unstable runtime states.

Optimized LLM Integration
  • Created `GeminiAI.js` Service: Integrated `@google/generative-ai` SDK utilizing the fast `gemini-2.5-flash` model.
  • Ayurvedic Custom Instruction Core: Prepopulated system instructions detailing Ayurvedic paradigms (Rasa, Virya, Vipaka, Ritucharya, Dinacharya).
  • Prakriti Profile Alignment: The model checks and adapts its entire guidance automatically based on the user's primary/secondary Dosha.
  • Strict History Sanitizer: Built `_sanitiseHistory()` to merge adjacent identical roles and conform to rigid user/model alternation constraints.
  • Exponential Backoff Retries: Integrated a 3x retry flow (2s, 4s, 8s delay) to bypass free-tier rate blocks.
💡 Stage 3 Architectural Purpose

Migrating to Gemini 2.5 Flash provided superior conversational quality and Ayurvedic medical reasoning. Direct RAG pipeline injections ensured the LLM stayed anchored in classical textbook facts.

STAGE 4

AI Footprint Consolidation & Stack Pruning

To prepare the system for lightweight production servers, the redundant, resource-intensive legacy frameworks from Stage 2 were removed.

Starting State (Stage 3)

Unused files (`RasaAI.js`, `HuggingFaceAI.js`, `LocalAI.js`) sat dormant, loading heavy packages. Environmental configuration files were bloated with deprecated credentials.

Consolidated Stack
  • File-System Deprecation: Erased `RasaAI.js`, `HuggingFaceAI.js`, and `LocalAI.js` from the codebase.
  • Refactored Orchestrator: Simplified `AIServiceManager.js` to manage only the `gemini` (Primary Cloud) and `custom` (Offline Local Fallback) providers.
  • Secured Configuration: Erased OpenAI API keys and Ollama connection parameters from `.env`.
💡 Stage 4 Architectural Purpose

This consolidation reduced host RAM requirements from 12GB to under 500MB, matching standard cloud host bounds and ensuring high performance.

STAGE 5

UI/UX Polish & SSR Build Stabilizers

The final phase polished frontend readability and addressed critical Server-Side Rendering (SSR) production build blocks.

Starting State (Stage 4)

The chatbot output unformatted plain text. The user did not know which AI model processed their query. The Next.js production build crashed due to browser window objects accessed globally during Node-side pre-rendering.

Polished & Stabilized Client
  • Markdown Integration: Created React markdown parser helper routines (`renderMarkdown`) to format recipes and diet schedules clearly.
  • Active Service Badging: Placed interactive sparkles badges (`✨ Gemini AI` or `⚙️ Rule Engine`) on chat bubbles for xAI auditing.
  • Next.js SSR Safety Guard: Protected token check in `AuthContext.js` with `typeof window !== 'undefined'` checks.
💡 Stage 5 Architectural Purpose

Formatting chatbot outputs ensures high readability for complex diet suggestions. The SSR check fixed fatal compilation errors, ensuring stable, successful production builds.
